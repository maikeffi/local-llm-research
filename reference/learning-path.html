Inference Engineer — 6-Month Learning Path

From zero GPU programming to measurable inference speedups on two platforms · anchored to the Moondream Inference Engineer role · 10–20 hrs/week · M4 Max local + cloud CUDA

Operating principle: every week ends with a number you measured yourself (tokens/sec, GB/s, kernel ms). Reading without benchmarking doesn't count as progress. The role's own bar: "You've made production inference workloads meaningfully faster, not just turned a benchmark green."

The six phases

Phase 1 · Performance Foundations (on your Mac)

Month 1 · ~60 hrs
  • Course A — The performance mental model: Horace He's "Go Brrrr" + kipply's Transformer Inference Arithmetic. Benchmark LLMs locally with mlx-lm on the M4 Max (lesson 0001 starts here).
  • Course B — C++ crash course: learncpp.com chapters 1–13 only (basics → pointers → functions). 30–45 min/day; stop there.
  • Course C — First parallel thinking: PMPP chapters 1–3 (read-only, concepts) + LeetGPU first easy challenges in-browser.
Outcome: You can predict a model's decode tokens/sec from memory bandwidth before running it, verify within 2× on your M4 Max, and explain the prefill/decode asymmetry. You can read simple C++/CUDA code without panic.

Phase 2 · Core CUDA

Month 2 · ~65 hrs
  • Course A — PMPP core: chapters 2–6 with every exercise coded (Colab T4 free tier is enough). Topics: thread hierarchy, global/shared memory, tiling, coalescing.
  • Course B — GPU MODE lectures 1–8: one per week alongside PMPP; Lecture 8 (performance checklist) is the keystone. Join the GPU MODE Discord this month and lurk the kernel channels.
  • Course C — Kernel reps: 3–4 LeetGPU problems/week: vector ops → reductions → softmax → tiled matmul.
Outcome: You can write a tiled shared-memory matmul from scratch, explain why it beats the naive version (memory traffic math), and identify uncoalesced access in someone else's kernel.

Phase 3 · Profiling & the Matmul Gauntlet

Month 3 · ~65 hrs
  • Course A — Nsight bootcamp: rent a real GPU a few hours/week (Lambda/RunPod, ~$0.50–2/hr) since Colab can't run Nsight Compute well. Profile every Phase-2 kernel; learn to read SOL, occupancy, and memory charts.
  • Course B — The SGEMM worklog: reproduce Simon Boehm's matmul optimization kernel-by-kernel (naive → coalesced → tiled → vectorized → warptiled). Keep your own written worklog with Nsight screenshots.
  • Course C — Roofline discipline: PMPP performance chapters; for each kernel state predicted vs achieved % of peak.
Outcome: A public blog post / repo: "Optimizing SGEMM from 2% to ~70% of cuBLAS" with profiler evidence at each step. This is portfolio artifact #1 and the single strongest interview asset at this stage.

Phase 4 · Inference Internals & Fusion

Month 4 · ~65 hrs
  • Course A — Build a tiny inference engine: implement GPT-2-class decoding in plain PyTorch/MLX: KV cache, greedy/temperature sampling, batching. Measure how each addition changes tokens/sec.
  • Course B — Attention & fusion: read the FlashAttention paper for the IO-aware idea; work through Triton tutorials (vector add → softmax → fused softmax → matmul) and write one fused kernel of your own (e.g. RMSNorm + residual).
  • Course C — Quantization, honestly: Lilian Weng's survey + hands-on INT4/INT8 with mlx-lm. Frame it as bandwidth reduction — measure why 4-bit ≈ 2× decode speed, and what it costs.
  • Course D — VLM specifics: run Moondream locally via mlx-vlm; profile where time goes (vision encoder vs projector vs text decode).
Outcome: You can draw the full latency budget of a VLM request, have written a working fused Triton kernel with before/after numbers, and can argue when quantization is the wrong answer — the exact anti-pattern Moondream screens for.

Phase 5 · Serving Systems & Platform Breadth

Month 5 · ~65 hrs
  • Course A — vLLM from the inside: read the architecture docs then source: scheduler, PagedAttention block manager, continuous batching. Run load tests; chart latency vs throughput vs batch size.
  • Course B — Apple Silicon deep dive: custom Metal kernels via MLX's extension API on your M4 Max; benchmark MLX vs llama.cpp on identical models; skim Core ML conversion for one small model. This is your second platform — the role explicitly requires multi-target thinking.
  • Course C — One vendor toolchain (concept level): TensorRT-LLM quickstart on a rented GPU; understand engines, graph optimization, and where it beats hand-rolled kernels. XLA/Neuron stay at reading level for interviews.
Outcome: You can explain PagedAttention from the code (not the blog post), have shipped one custom Metal kernel, and can compare the same model's performance across CUDA and Apple Silicon with real numbers — portfolio artifact #2.

Phase 6 · Capstone & Going Public

Month 6 · ~60 hrs
  • Capstone: pick ONE real optimization target and make it meaningfully faster, end to end. Best options: (a) speed up Moondream inference on Apple Silicon (mlx-vlm path) with profiling evidence, or (b) contribute a measured kernel/perf improvement upstream to mlx-lm, mlx-vlm, or llama.cpp.
  • Write-up: a thorough public post — baseline, profile, hypothesis, fix, numbers, what didn't work. Honest negative results included.
  • Community: post the worklog in GPU MODE Discord and r/LocalLLaMA; enter a GPU MODE kernel competition if one is running; iterate on feedback.
  • Interview prep: rehearse the spine questions — roofline math, why decode is memory-bound, how FlashAttention works, KV cache sizing, fusion trade-offs ("performance vs code clarity" is literally in the job description).
Outcome: A portfolio that answers the role's first screen — "made a real inference workload meaningfully faster" — with public, reproducible evidence on two platforms, plus a community footprint where these companies actually recruit.
