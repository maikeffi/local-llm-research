Inference Performance Glossary

Canonical vocabulary for this workspace · grows with each lesson · terms introduced in lessons 0001–0002 so far

Core quantities

FLOPs / FLOP/s
Floating-point operations (a count) vs floating-point operations per second (a rate). A matmul of (M×K)·(K×N) costs 2·M·K·N FLOPs — the 2 is one multiply + one add.
Memory bandwidth (GB/s)
How fast bytes stream between memory and compute units. Your M4 Max: 546 GB/s nominal; your measured number from lesson 0002 is the one to use.
Arithmetic intensity (AI)
FLOPs performed per byte moved. The single number that locates a workload on the roofline.
AI = FLOPs / bytes moved   (FLOPs per byte)
Tokens/sec (tok/s)
The user-facing inference metric. Always specify which: prompt (prefill) tok/s or generation (decode) tok/s — they differ by an order of magnitude or more.

The roofline model

Roofline
Chart of achievable FLOP/s vs arithmetic intensity. Two ceilings: a slanted line (bandwidth × AI) and a flat line (peak compute). A workload's AI determines which ceiling caps it.
Ridge point
The AI where the two ceilings meet. Below: memory-bound. Above: compute-bound.
ridge = peak FLOP/s ÷ peak bytes/s   (FLOPs per byte)
Memory-bound (bandwidth-bound)
Speed limited by byte movement; compute units idle. Fix by moving fewer bytes (quantization, fusion) or reusing each byte more (batching, tiling). More TFLOPs won't help.
Compute-bound
Speed limited by math throughput; the memory system keeps up. Fix with faster math (tensor cores, lower precision) or fewer FLOPs (better algorithms).
Overhead-bound
Limited by neither — time goes to launch latency, Python, framework bookkeeping. Common for tiny kernels; fix with fusion, graphs, bigger batches. (Horace He's third regime; becomes central in month 4.)

LLM inference anatomy

Prefill
Processing the whole prompt in one parallel pass. Matrix–matrix work, high AI → typically compute-bound. Determines time-to-first-token.
Decode
Generating output one token at a time, each requiring a full pass over the weights. Matrix–vector work, AI ≈ 1–4 → memory-bound. Determines generation tok/s.
KV cache
Stored attention keys/values for all previous tokens, so each new token attends without recomputation. Grows linearly with context length; its reads are extra memory traffic that slowly erodes decode speed.
Quantization
Storing weights in fewer bits (8/4-bit vs fp16). In roofline terms: a bandwidth optimization — fewer bytes per token, so memory-bound decode speeds up roughly proportionally. One tool among many, never the whole strategy.
Batching
Decoding several requests together so each weight byte read serves the whole batch — multiplies AI by batch size. Past batch ≈ ridge ÷ AI, decode turns compute-bound. Foundation of serving systems (month 5).

Measurement discipline

Warmup
Discarded initial iterations that absorb one-time costs (compilation, allocation, cache fill) so timings reflect steady state.
Lazy evaluation / synchronization
MLX builds graphs lazily; CUDA launches kernels asynchronously. Timing without forcing completion (mx.eval, cudaDeviceSynchronize) measures bookkeeping, not work. The classic benchmarking bug.
Nominal vs achieved
Spec-sheet peak vs what your benchmark sustains. Kernels are judged as % of achievable peak, not marketing peak.
