The GPU Leap: Same Task, Thousands of Threads

The one thing this lesson teaches: a GPU wins at memory-bound work not by being faster, but by keeping thousands of memory requests in flight at once. You'll run the same array-sum you ran in C++ on your M4 Max's GPU, watch it approach the full memory bus you measured in lesson 2, and confirm the "you need ~20× more parallelism" conclusion you derived on the CPU.

1 · The setup — and why your Mac makes this experiment perfect

In lesson 4 you summed a 1 GiB array on one CPU core and hit ~26 GB/s — about 5% of your 546 GB/s bus — because a single core can only keep a handful of memory requests outstanding at a time. You concluded you'd need ~20× more parallel requests to fill the bus. Let's test that on the engine built for it.

The unified-memory gift. On Apple Silicon, the CPU and GPU share the same physical memory on the same bus — there's no copying data to a separate "GPU card." So when you sum the array on the CPU and then on the GPU, both read identical bytes over an identical 546 GB/s pipe. The only variable that changes is how much parallelism the engine brings. That's a controlled experiment most GPU programmers can't run cleanly — for you it's one line of code.

2 · The GPU mental model, in three sentences

  • Data parallelism: summing 268M numbers is split across tens of thousands of GPU threads — each adds up a small chunk, then the partial results are combined in a tree (a reduction).
  • Memory-level parallelism: with that many threads, hundreds of memory requests are in flight simultaneously, so the bus stays full — this, not faster arithmetic, is the whole advantage on a memory-bound task.
  • Latency hiding: when some threads stall waiting for memory (~100 ns), the scheduler instantly runs others that are ready, so the expensive memory latency you met in lesson 3 is simply overlapped away.

Your M4 Max GPU has 40 cores, each running many threads — together tens of thousands in flight. A single CPU core, by contrast, has one out-of-order window keeping ~10 requests going. That ratio is the experiment.

3 · The experiment

Step 1 — gpu_sum.py

import time
import mlx.core as mx

N = 1 << 28                       # 268,435,456 floats = 1 GiB, same as your C++ test
x = mx.ones((N,), dtype=mx.float32)
mx.eval(x)                        # materialize the array BEFORE timing

def bench(f, iters=50, warmup=10):
    for _ in range(warmup):
        mx.eval(f())              # discard compile + warmup runs
    t0 = time.perf_counter()
    for _ in range(iters):
        mx.eval(f())              # force the lazy graph to actually run
    return (time.perf_counter() - t0) / iters

t = bench(lambda: mx.sum(x))
bytes_moved = N * 4              # read every float once
print(f"GPU sum: {t*1e3:.3f} ms | "
      f"{bytes_moved / t / 1e9:.0f} GB/s | "
      f"{t / N * 1e9:.4f} ns/element")
python3 gpu_sum.py

It sums the same 1 GiB of floats on the GPU and reports time, achieved bandwidth, and ns per element — the same metrics you used on the CPU, so the comparison is apples to apples.

Predict before running. Your CPU single-core floor was 0.153 ns/element (26 GB/s). For the GPU: (a) what ns/element, (b) what GB/s — and how close to your lesson-2 ceiling of 436 GB/s do you expect it to land?
What you should see, and why

Expect roughly 2.5–3.5 ms, ~300–436 GB/s, and ~0.009–0.013 ns/element — i.e. a large fraction of your measured bus peak, and on the order of 12–17× faster per element than one CPU core. That multiple is not a coincidence: it's the ~20× parallelism you predicted in lesson 4, made real. The GPU isn't doing the additions faster (addition was never the bottleneck here) — it's keeping enough loads in flight to saturate the same memory bus the lone CPU core could only 5%-fill.

4 · The clean A/B — flip one line

Step 2 — same code, same data, CPU vs GPU

MLX can run the identical operation on either engine. Add this to compare directly:

for dev in (mx.cpu, mx.gpu):
    mx.set_default_device(dev)
    t = bench(lambda: mx.sum(x))
    name = "CPU" if dev == mx.cpu else "GPU"
    print(f"{name}: {t*1e3:.3f} ms | {N*4 / t / 1e9:.0f} GB/s")

Same array in the same memory, same one line of math — only the engine differs. The GB/s gap you see is the parallelism gap, isolated. (MLX's CPU backend uses multiple cores, so it'll beat your single-core C++ number but still fall well short of the GPU — even all your CPU cores together can't match the GPU's memory-request throughput on this workload.)

5 · What this means for inference

  • This is why decode runs on GPUs. You showed in lessons 1–2 that LLM decode is memory-bound. Memory-bound work is exactly what the GPU's thousands-of-requests-in-flight design accelerates — so generation throughput lives or dies on memory bandwidth and how fully you use it.
  • But one request can't fill a big GPU. A single decode stream is like a small array — too little parallel work to keep a datacenter GPU's threads busy, so it wastes most of the chip. The fix is the same lever from lesson 2: batch many requests together to supply enough parallel work. This is the seed of continuous batching (vLLM, month 5).
  • Soon you'll write the threads yourself. Today MLX created and scheduled the threads for you. In month 2 (CUDA) you will decide how work maps onto threads, how they read memory (coalesced — your stride lesson!), and how partial sums combine. Every concept you just watched, you'll soon control by hand.

6 · Self-check

  1. A colleague says "the GPU summed faster because its cores do addition faster." Why is that the wrong explanation for this workload?
  2. You measured GPU ≈ 16× your single CPU core. Connect that number to your lesson-4 conclusion.
  3. Why does Apple's unified memory make the CPU-vs-GPU comparison unusually clean compared to a typical NVIDIA setup?
  4. Inference transfer: why does a single user's decode request underutilize a large GPU, and what's the standard remedy?
Answers
  1. The task is memory-bound — addition was never the limiter. The GPU wins by keeping far more memory requests in flight (memory-level parallelism), saturating the bus the CPU core could only 5%-fill.
  2. Lesson 4 said one core gets ~1/20th of the bus and you'd need ~20× the parallelism to fill it; the GPU supplies that parallelism, so it runs ~the same multiple faster.
  3. No host↔device copy and one shared bus mean both engines read identical bytes at the same bandwidth — the only variable is parallelism. On an NVIDIA box you'd also pay PCIe transfer time and use separate memory, muddying the comparison.
  4. One stream is too little parallel work to keep all GPU threads busy, so most of the chip idles; batch many requests so the GPU has enough independent work to saturate.
Your win: you ran the same workload on a CPU core and a GPU over the same memory bus, and watched raw parallelism turn 5% bus utilization into near-peak — confirming, by measurement, the conclusion you derived on the CPU. You now understand why GPUs dominate inference, not as a slogan but as a number you produced. Next you'll start telling the GPU's threads what to do.
