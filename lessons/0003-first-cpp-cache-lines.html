First C++, and the Cache Line You Can Measure

The one thing this lesson teaches: memory is delivered in fixed-size chunks called cache lines, so how you walk through memory matters as much as how much you read. You'll write your first C++ program, use it to experimentally discover your M4 Max's cache line size, and meet the concept that becomes CUDA's "memory coalescing" in month 2.

1 · Why C++ at all

Python tells a runtime what to do; C++ compiles to the actual machine instructions your CPU executes. Kernels — CUDA or Metal — are written in C++ dialects because at kernel level you decide every memory access. This track teaches the working subset CUDA needs (per learncpp.com ch. 1–13), not the whole language.

Step 1 — Compile something

Your Mac ships clang. Check, then create and compile:

clang++ --version        # if missing: xcode-select --install
// hello.cpp
#include <cstdio>

int main() {
    int bytes_per_float = 4;
    std::printf("a float is %d bytes\n", bytes_per_float);
    return 0;
}
clang++ -O2 -std=c++17 hello.cpp -o hello && ./hello

Three things Python never showed you: int bytes_per_float declares a typed variable (the compiler must know sizes — types ARE memory layout here); main is the fixed entry point; -O2 asks the optimizer to work hard — always benchmark with -O2, unoptimized C++ numbers are meaningless.

2 · The mental model: memory arrives by the line, not by the byte

When your program reads one 4-byte float, the memory system doesn't fetch 4 bytes. It fetches the entire cache line containing it — a fixed chunk (64 or 128 bytes depending on the CPU) — into fast cache. Read the neighboring float next and it's already there, nearly free. Skip far ahead and you pay for a whole new line while using 4 bytes of it.

Ask your machine what it claims, so the experiment has a hypothesis to test:

sysctl hw.cachelinesize

3 · The experiment: discover the line size yourself

Step 2 — stride.cpp

// stride.cpp — time per element-touch as stride grows
#include <chrono>
#include <cstdio>
#include <vector>

int main() {
    const size_t N = 1 << 28;               // 268M floats = 1 GiB
    std::vector<float> data(N, 1.0f);        // contiguous block of N floats

    for (size_t stride = 1; stride <= 128; stride *= 2) {
        double sum = 0.0;
        auto t0 = std::chrono::steady_clock::now();
        for (size_t i = 0; i < N; i += stride)
            sum += data[i];
        auto t1 = std::chrono::steady_clock::now();

        double ns = std::chrono::duration<double, std::nano>(t1 - t0).count();
        size_t touches = N / stride;
        std::printf("stride %4zu (%5zu B apart) | %6.3f ns/touch | sum=%.0f\n",
                    stride, stride * sizeof(float), ns / touches, sum);
    }
}
clang++ -O2 -std=c++17 stride.cpp -o stride && ./stride

What it does: touches every stride-th float of a 1 GiB array. Each row does fewer touches, but each touch lands farther from the last. We print sum so the optimizer can't delete the loop (a real benchmarking discipline — dead code runs in 0 ns).

Predict before running: ns/touch at stride 1 vs stride 32 (= 128 bytes apart) — same, 2× worse, or ~10× worse? And after the knee, will doubling the stride again keep making each touch slower?
What you should see, and why

Time per touch climbs as stride grows — each touch shares less of its cache line with the previous one — then flattens at the stride where one touch = one new line (stride × 4 bytes ≥ line size). That knee IS your cache line size, measured. Compare it to sysctl's claim. Past the knee, every touch misses regardless, so doubling stride barely changes ns/touch (until far bigger strides hit paging effects — month 2 territory).

Bonus reading of the same numbers: at stride 1 you stream 1 GiB sequentially. Compute your single-core bandwidth: 4 bytes ÷ (ns per touch) GB/s. It will land well below your 436 GB/s — one CPU core physically cannot drive the whole memory bus. Bandwidth is a shared, per-engine resource; the GPU number you measured in lesson 0002 needed thousands of parallel accesses in flight. Hold that thought — it's exactly why GPUs look the way they do.

5 · Self-check

  1. Your knee appeared at stride k. Write the cache line size formula and your measured value.
  2. Roofline vocabulary (lesson 0002): summing floats is 1 add per 4 bytes → AI = 0.25. Is this loop compute- or memory-bound, and does this experiment depend on that?
  3. A colleague benchmarks without -O2 and without printing sum. Name both bugs.
  4. Transfer: a model stores its weight matrix row-by-row, but a kernel reads it column-by-column. What does this lesson predict?
Answers
  1. line = k × sizeof(float) = k × 4 bytes. Most Apple Silicon reports 128 B (knee at stride 32) — trust your measurement.
  2. Deeply memory-bound (AI 0.25 vs ridge ~33). Yes, the experiment depends on it: because compute is negligible, ns/touch is a clean probe of pure memory behavior.
  3. Without -O2 you time the unoptimized interpreter-grade code; without using sum the optimizer may delete the loop entirely and you time nothing.
  4. Column reads are strided by the row length — every access pays for a full line and uses 4 bytes, so effective bandwidth collapses (often ~10–30×). Fixing exactly this via access-pattern changes is CUDA "coalescing," month 2's centerpiece.
Your win: you wrote, compiled, and ran C++ for the first time — and used it to measure a real microarchitectural constant of your machine. You now hold the third leg of the performance tripod: bandwidth (lesson 2's ceiling), arithmetic intensity (lesson 2's x-axis), and access pattern (today). Every kernel optimization you'll ever do manipulates one of these three.
