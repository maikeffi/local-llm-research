Reading Your Benchmark: ns, Strides & Parallel Accumulators

The one thing this lesson teaches: how to read a memory benchmark line by line — what a nanosecond and a stride actually are, what every line of the C++ did, and why a "parallel accumulator" rewrite speeds things up in one regime but not another. The deep takeaway: an optimization only helps where it attacks the binding bottleneck.

1 · What is a nanosecond (ns)?

A nanosecond is one billionth of a second: 1 ns = 0.000000001 s = 10⁻⁹ s. It's the natural unit for "how long did one tiny operation take," because modern hardware operates at this scale:

EventRoughly
One M4 P-core clock tick (~4.5 GHz)~0.22 ns
Reading a value already in L1 cache~1 ns
Reading a value that misses all caches (DRAM)~80–100 ns
Light traveling across your desk~1 ns per 30 cm

"ns/touch" in your output = the average nanoseconds spent per element the loop accessed. Lower is faster. It's a per-operation cost, which is what lets you compare rows that did different amounts of work.

2 · What is a stride?

A stride is the gap between successive accesses, measured in elements. Walking an array:

stride 1:  index 0, 1, 2, 3, 4, ...      // every element (consecutive)
stride 4:  index 0, 4, 8, 12, ...        // every 4th element
stride 32: index 0, 32, 64, 96, ...      // every 32nd element

Since a float is 4 bytes, "stride 32" means each access lands 32 × 4 = 128 bytes past the last — that's the "B apart" column. We sweep the stride (doubling it each row) to discover how far apart accesses have to be before the cache stops helping us. That distance reveals the cache line size.

3 · The original code, line by line

#include <chrono>     // timing tools (clocks, durations)
#include <cstdio>     // std::printf for output
#include <vector>     // std::vector, a growable array

int main() {
    const size_t N = 1 << 28;            // 1<<28 = 2^28 = 268,435,456 floats (1 GiB)
    std::vector<float> data(N, 1.0f);    // one contiguous block; every element = 1.0

    for (size_t stride = 1; stride <= 128; stride *= 2) {   // 1,2,4,...,128
        double sum = 0.0;
        auto t0 = std::chrono::steady_clock::now();         // start the stopwatch
        for (size_t i = 0; i < N; i += stride)              // touch every stride-th float
            sum += data[i];                                 // read it, add to sum
        auto t1 = std::chrono::steady_clock::now();         // stop the stopwatch

        double ns = std::chrono::duration<double, std::nano>(t1 - t0).count();
        size_t touches = N / stride;                        // how many elements we read
        std::printf("stride %4zu (%5zu B apart) | %6.3f ns/touch | sum=%.0f\n",
                    stride, stride * sizeof(float), ns / touches, sum);
    }
}

The pieces that matter

  • 1 << 28 — a bit shift: take 1 and move it 28 binary places left = 2²⁸. A common C++ idiom for "a power-of-two size."
  • size_t — an unsigned integer type sized to hold any array index on your machine (64-bit here). The standard type for sizes and indices.
  • std::vector<float> data(N, 1.0f) — allocates N floats in one contiguous run of memory and fills them with 1.0. Contiguity is the whole point: it's what makes strides meaningful.
  • i += stride — the inner loop's step. This single line is the experiment.
  • steady_clock::now() — a monotonic timer; subtract two readings to get elapsed time. duration<double, std::nano> converts that to nanoseconds.
  • ns / touches — normalize: total time ÷ number of accesses = ns per touch. Essential, because each row does a different number of touches.
Why the sum column is a built-in correctness check

Every element is 1.0, so summing k touches must give exactly k. Your stride-1 row: sum = 268435456 = N. Stride 2: 134217728 = N/2. Each row's sum equals its touch count — proof the loop ran correctly and that the optimizer didn't silently delete it. Printing a result you can verify is a benchmarking habit worth keeping forever.

The story in one sentence: your cache line is 128 bytes. Up to stride 16 (64 B apart), multiple of your touches still land inside the same 128-byte line that was already fetched — cheap. At stride 32 (128 B apart) each touch lands in its own line, so every single touch pays for a 128-byte fetch to use just 4 bytes — a 4.3× jump. Beyond that it creeps up as the prefetcher and TLB lose effectiveness.

The hidden waste at stride 32+

At 2.245 ns/touch you move 128 bytes to use 4. Bytes actually moved: 128 B ÷ 2.245 ns ≈ 57 GB/s — but useful throughput is 4 B ÷ 2.245 ns ≈ 1.8 GB/s. You're wasting ~97% of the bandwidth you're paying for. Eliminating exactly this waste — making every byte fetched a byte used — is what "memory coalescing" means on a GPU (month 2).

5 · The merged code: naive vs parallel, at every stride

This version times two ways of summing the same elements at each stride, so you can compare them directly:

#include <chrono>
#include <cstdio>
#include <vector>

using clk = std::chrono::steady_clock;
static double ns_per(double ns, size_t touches) { return ns / touches; }

// ONE accumulator: each add waits for the previous one (a serial chain)
static double time_naive(const std::vector<float>& d, size_t stride) {
    const size_t N = d.size();
    double sum = 0.0;
    auto t0 = clk::now();
    for (size_t i = 0; i < N; i += stride) sum += d[i];
    auto t1 = clk::now();
    volatile double sink = sum; (void)sink;          // keep the optimizer honest
    return ns_per(std::chrono::duration<double,std::nano>(t1-t0).count(), N/stride);
}

// FOUR accumulators: four independent chains the CPU can overlap
static double time_parallel(const std::vector<float>& d, size_t stride) {
    const size_t N = d.size();
    float s0=0, s1=0, s2=0, s3=0;
    const size_t step = stride * 4;
    auto t0 = clk::now();
    size_t i = 0;
    for (; i + 3*stride < N; i += step) {            // same indices as naive...
        s0 += d[i];            s1 += d[i + stride];
        s2 += d[i + 2*stride]; s3 += d[i + 3*stride]; // ...just 4 separate sums
    }
    for (; i < N; i += stride) s0 += d[i];           // leftover tail
    auto t1 = clk::now();
    volatile double sink = double(s0)+s1+s2+s3; (void)sink;
    return ns_per(std::chrono::duration<double,std::nano>(t1-t0).count(), N/stride);
}

int main() {
    const size_t N = 1 << 28;
    std::vector<float> data(N, 1.0f);
    std::printf("stride | bytes |  naive |    par | speedup\n");
    for (size_t stride = 1; stride <= 128; stride *= 2) {
        time_naive(data, stride);                    // warmup (discarded)
        double a = time_naive(data, stride);
        double b = time_parallel(data, stride);
        std::printf("%6zu |%5zu B | %5.3f | %5.3f | %5.2fx\n",
                    stride, stride*sizeof(float), a, b, a/b);
    }
}
clang++ -O2 -std=c++17 stride2.cpp -o stride2 && ./stride2

What actually changed, and why it's a fair test

Both functions touch the exact same elements in the same order — index 0, stride, 2·stride, 3·stride, 4·stride, … The only difference is the number of running totals:

  • Naive: one sum. Add #2 cannot start until add #1 finishes — a single chain. The CPU is forced to wait ~3 cycles (FP-add latency) per element, even though it could do more.
  • Parallel: four sums (s0..s3) that don't depend on each other, so the CPU's out-of-order engine runs them at the same time. Same work, four lanes instead of one.

This trick has a name you'll meet again: exposing instruction-level parallelism (ILP). (Note: clang won't do it for you on floats — reordering a+b+c can change rounding, so the compiler leaves your serial chain alone unless you rewrite it.)

Predict before you run. The parallel version breaks the FP-add chain. In which rows will it help most?
(a) the flat rows (stride 1–16), (b) the cliff rows (stride 32–128), or (c) equally everywhere?
Reveal — and the deep lesson

(a) the flat rows. There the FP-add chain was the bottleneck (~0.5 ns ≈ 2.4 cycles, pure compute latency), so adding lanes should drop stride-1 toward ~0.15–0.25 ns — a real 2–3× win. In the cliff rows, the bottleneck is the 128-byte fetch per touch; the add chain is already hidden under a ~2.2 ns memory wait, so four accumulators barely move the number (~1.0–1.2×).

This is the whole point of the lesson: the same optimization is a big win or a no-op depending on which constraint is binding. Speeding up compute does nothing when you're memory-bound; reducing memory traffic does nothing when you're compute-bound. Every hour you'll ever spend optimizing inference starts with the question you just answered experimentally: what is actually holding this back right now?

6 · Why this matters for your future

  • This IS GPU coalescing, previewed. On a GPU, the 32 threads of a "warp" read memory together. If they read consecutive addresses (the stride-1 case), one fetch serves all 32 — fast. If they read scattered addresses (the stride-32 case), the hardware issues many fetches — slow. Uncoalesced access is the #1 beginner CUDA performance bug, and you've now felt it on your own CPU.
  • The flat-then-cliff shape is universal. You'll see the same curve crossing every boundary of the memory hierarchy: registers → L1 → L2 → DRAM on a CPU, and registers → shared memory → global memory on a GPU. Recognizing the shape tells you which level you just fell out of.
  • Bottleneck-first thinking is the job. Profilers (Nsight, month 3) exist to answer "what's binding?" automatically. You're learning to answer it by hand first, which is why the profiler output will mean something to you.
  • Benchmarking hygiene transfers verbatim: warm up, compile with -O2, defeat dead-code elimination, normalize to per-operation cost, print a verifiable result. You'll use every one of these on GPU kernels unchanged.

The prediction held — and the gradient (3.2× → 1.84× → 1.09× → 1.00×) is the real prize. It's a continuous readout of how much of each touch was compute you could overlap. As stride grows inside the 128 B line, memory cost per touch rises; in the naive loop it hid under the FP-add latency until it grew big enough to dominate. The speedup column literally measures "what fraction of this was compute-bound." Nsight Compute's SM-busy vs memory-throughput bars (month 3) are this same meter, automated.

Two numbers worth keeping

  • Real single-core memory floor: stride-1 parallel 0.153 ns/touch → 4 B ÷ 0.153 ns ≈ 26 GB/s. That's ~5% of the 546 GB/s bus — one core flat-out gets ~1/20th of memory. Filling the bus needs ~20× that parallelism; a GPU supplies it with thousands of threads. (The naive 0.490 ns was latency-bound and understated this — the parallel rewrite reveals the truth.)
  • You crossed the roofline ridge. Arithmetic per byte fetched from DRAM: stride 1 ≈ 32 adds/line (right of ridge → compute-bound); stride 32 ≈ 1 add/line, 32× lower (left of ridge → memory-bound). The sweep slid one workload across lesson 2's ridge, and the speedup confirms compute fixes only help on the compute-bound side.
Optional curiosity — push the floor further

Try 8 accumulators (s0..s7). If stride-1 parallel drops below 0.153 ns, you were still compute-throughput-bound and found more headroom; if it stalls at ~0.153, you've hit the genuine single-core memory/load-throughput wall. Either answer is a clean result — that's the game.

8 · Self-check

  1. In your own words: why is "ns/touch" a fairer comparison than total loop time across the rows?
  2. Your cliff is at stride 32 = 128 B. If a friend's machine cliffs at stride 16, what's their cache line size?
  3. You run the merged code and stride-1 speedup is 2.6×, but stride-64 speedup is 1.05×. Explain both in one sentence each.
  4. Transfer: a GPU kernel where neighboring threads read array elements 1024 apart — fast or slow, and which row of your table is the analogy?
Answers
  1. Each row touches a different number of elements (N/stride), so total time isn't comparable; dividing by touches gives the cost of one access, which is the thing we're actually probing.
  2. 16 × 4 B = 64-byte cache line (typical of many x86 CPUs).
  3. Stride-1 was compute/latency-bound, so breaking the FP chain helped a lot; stride-64 was memory-bound (one fetch per touch), so more accumulators couldn't help — the wait was for memory, not arithmetic.
  4. Slow — scattered reads, one fetch per thread, ~97% wasted bandwidth. It's the stride-32+ cliff rows. The fix is the GPU version of "make threads read consecutively" = coalescing.
Your win: you can now read a memory benchmark like a practitioner — you know what a ns and a stride are, what every line of the code does, how to spot the cache-line cliff in raw numbers, and why an optimization's value depends entirely on which bottleneck is binding. That last idea is the spine of the entire profession.
