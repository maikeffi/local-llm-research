Lesson 0001 · ~45 minutes · Phase 1

Your First Inference Benchmark

The one thing this lesson teaches: LLM text generation is limited by memory bandwidth, not compute — and you can predict a model's speed with arithmetic before you ever run it. This is the foundational mental model of the entire job.

1 · Run a model and get your numbers

Step 1 — Install MLX's LLM runner

python3 -m venv ~/.venvs/infer && source ~/.venvs/infer/bin/activate
pip install mlx-lm

mlx-lm is Apple's framework for running LLMs natively on your M4 Max's GPU.

Step 2 — Generate text and read the stats

mlx_lm.generate \
  --model mlx-community/Llama-3.2-3B-Instruct-4bit \
  --prompt "Explain why GPUs are good at matrix multiplication." \
  --max-tokens 200

It downloads ~1.8 GB the first time. At the end MLX prints two numbers. Write them down:

  • Prompt: ... tokens-per-sec — the prefill speed (processing your input)
  • Generation: ... tokens-per-sec — the decode speed (producing output, one token at a time)
Predict before you peek: which number will be bigger, and roughly by how much — 2×? 10×? 50×? Commit to a guess before running.

2 · Why the two numbers are so different

Every generated token requires reading every weight of the model from memory. Your 4-bit 3B model is ~1.8 GB, so producing one token moves ~1.8 GB through the memory system. Prefill is different: all prompt tokens are processed together in one pass, so one trip through the weights is shared across many tokens.

That's the fundamental asymmetry (kipply's inference arithmetic, Horace He's "Go Brrrr"):

  • Prefill is compute-bound — limited by how fast the chip multiplies matrices.
  • Decode is memory-bandwidth-bound — limited by how fast weights stream from memory.

3 · Predict your machine's speed limit

The napkin math

Your M4 Max has ~546 GB/s memory bandwidth. If every token needs one full pass over ~1.8 GB of weights:

ceiling ≈ bandwidth / model size
        ≈ 546 GB/s ÷ 1.8 GB ≈ ~300 tokens/sec

Real systems hit 50–80% of this ceiling (KV cache reads, kernel overhead, non-weight memory traffic). So expect roughly 150–240 tok/s decode.

Check: was your measured decode speed within that band?

If yes — you just did your first roofline-style analysis, the exact reasoning Nsight profiling formalizes later. If it was far off, tell the teaching agent your number; diagnosing the gap is a great follow-up session.

4 · Prove it with an experiment

Run the 8-bit version of the same model

mlx_lm.generate \
  --model mlx-community/Llama-3.2-3B-Instruct-8bit \
  --prompt "Explain why GPUs are good at matrix multiplication." \
  --max-tokens 200

The 8-bit model is ~2× larger in bytes (~3.4 GB) but does the same math.

Predict first: if decode is truly bandwidth-bound, what should happen to generation tok/s when the weights double in size?
Reveal

Decode speed should drop to roughly half — twice the bytes per token through the same pipe. If you observed ~½, you've experimentally confirmed decode is memory-bound. This is also why quantization speeds up decode: it's a bandwidth optimization, not a compute one — and why Moondream flags people who think quantization is the only lever.

5 · Quick self-check

  1. A friend says "my GPU has huge TFLOPs, so generation will be fast." What's wrong with that reasoning for decode?
  2. You double the prompt length. Which number suffers more — prefill time or decode tok/s? Why?
  3. Estimate: decode ceiling for an ~8 GB model on your machine?
Answers
  1. Decode is bandwidth-bound: TFLOPs sit idle while weights stream from memory. Bandwidth, not compute, sets the ceiling.
  2. Prefill time grows with prompt length (more tokens to crunch — compute-bound). Decode tok/s barely changes at first (weights dominate traffic), degrading slowly as the KV cache grows.
  3. ≈ 546 / 8 ≈ 68 tok/s ceiling → ~40–55 tok/s realistic.
Your win: you measured real inference numbers on your own hardware, predicted them from first principles, and confirmed the prediction with an experiment. That predict → measure → explain loop is the day-to-day core of inference engineering.
